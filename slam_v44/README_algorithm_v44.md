# 自己实现的 SLAM 算法 + 触点导航 v44

## 一、整体说明

`slam_v44` 是在原版基础上修改后的二维激光 SLAM + 触点导航实验包。它订阅老师包发布的雷达数据和 odom 数据，把雷达数据转换成二维点；启动时固定 `map` 坐标系原点；后续用 odom 估计机器人相对启动点的移动，再用 scan matching 在 odom 预测位置附近做小范围修正；最后更新栅格地图，并发布地图、位姿、轨迹和 TF。

核心思路是：

```text
启动点固定为 map 原点
odom 估计机器人相对启动点的移动
scan matching 只做小范围修正
地图始终写入固定 map 坐标系
```

订阅的数据：

```text
/slamware_ros_sdk_server_node/scan
/slamware_ros_sdk_server_node/odom
```

发布的数据：

```text
/slam_v44/map        SLAM 建出来的二维栅格地图
/slam_v44/pose       机器狗在 map 坐标系下的位置
/slam_v44/slam_path  机器狗走过的轨迹
TF                   默认发布 map -> odom 修正
```

## 二、SLAM 算法部分

SLAM 部分在代码里叫：

```text
slam_v44_node
```

### Step1：处理雷达数据

节点订阅 `/slamware_ros_sdk_server_node/scan`，读取一圈 `LaserScan` 数据，把每个有效雷达距离点根据角度转换成机器人局部坐标系下的二维点：

```text
range + angle -> (x, y)
```

也就是把一圈雷达变成多个局部坐标点。

### Step2：固定启动点作为 map 原点

v44 增加了 odom。节点启动后收到第一帧 odom 时，会把这一帧 odom 锁定为启动参考点。

也就是说：

```text
机器人启动位置 = map 坐标系下的 (0, 0, 0)
```

后续机器狗移动时，代码会根据当前 odom 和启动 odom 的差值，计算机器人相对启动点移动了多少。

### Step3：odom 估计当前位姿

v44 不再单纯依赖雷达匹配估计机器人移动，而是先用 odom 预测当前位姿：

```text
relative_odom = 当前 odom 相对启动 odom 的变化
predicted_pose = 启动 map 位姿 + relative_odom
```

这样机器人移动后，系统知道它相对于最开始的 `(0,0)` 走到了哪里。

### Step4：scan matching 小范围修正

拿到 odom 预测位姿后，scan matching 只在这个预测位姿附近搜索一小圈候选 pose。

匹配过程可以理解为：

代码以 odom 预测出来的当前位姿为中心，在附近生成多个候选位姿 `(x, y, theta)`。每个候选位姿都代表“假设机器狗现在在这里”。然后把当前雷达点按照这个候选位姿投影到 `map` 坐标系里，再检查这些投影点附近是否已经存在地图障碍物。

如果某个候选位姿下，大量雷达点都能落在或接近已有障碍物，说明当前雷达和旧地图对得比较准，这个候选位姿的分数就高。最后选择分数最高、且超过最低匹配阈值的候选位姿，作为机器狗当前修正后的位姿。

### Step5：更新栅格地图

地图使用二维 `OccupancyGrid` 栅格地图。

每条雷达射线：

```text
机器人当前位置 -> 雷达终点
```

中间经过的格子标记为空闲，雷达终点标记为障碍物。多次观测会累加占据概率，当某个格子的占据概率超过阈值后，它就会变成障碍物格子。

红点显示节点会从 `/slam_v44/map` 中筛选障碍边界格子，并发布到：

```text
/slam_v44/persistent_obstacle_markers
```

这个红点主要用于 RViz 可视化，不是导航真正使用的地图本体。

## 三、v44 对原问题的修正

原版的问题是：

位姿主要靠雷达 scan matching 推出来，缺少 odom 提供连续运动预测。机器人走远后，scan matching 容易跟不上真实运动，导致当前雷达点没有稳定投影到同一个固定 `map` 坐标系里。现象上就会看起来像障碍物围着机器人当前位置反复生成，地图和红点容易堆叠混乱。

v44 的修正是：

```text
启动点固定为 map 原点
odom 估计机器人相对启动点的移动
scan matching 只做小范围修正
地图始终写入固定 map 坐标系
```

这样机器人移动后，雷达点会根据 odom 推算出的全局位姿写入地图，不再完全依赖雷达匹配估计长距离移动。理论上可以明显减少障碍物被错误堆叠到原点附近的问题，也能让导航拿到更稳定的机器人位姿。

不过 v44 仍然不是完整工业级 SLAM，它还没有真正的回环检测和图优化，所以走很远或者绕一大圈回来时，仍可能有累计漂移。

## 四、导航部分

导航部分在代码里叫：

```text
slam_v44_navigation_node
```

导航思路是：

先接收地图、目标点和机器狗当前位置，然后用 A* 规划路径，根据路径生成速度指令。如果前方雷达检测到障碍物太近，就停止或减速。

发布的话题：

```text
/slam_v44/global_path      A* 规划出来的全局路径
/slam_v44/debug_cmd_vel    调试用速度指令
```

真正上机让机器狗动之前，需要新开终端执行：

```bash
rosrun unitree_legged_real twist_sub
```

确认 debug 速度正常后，再把速度输出切到：

```text
/cmd_vel
```

## 五、导航流程

### Step1：RViz 点击目标点

在 RViz 里点击 `2D Nav Goal`，目标点会发布到：

```text
/move_base_simple/goal
```

导航节点收到后保存目标点。

### Step2：获取机器人当前位置

导航节点优先通过 TF 查询机器人当前位置：

```text
map -> odom -> base_link
```

如果 TF 不可用，也可以使用 `/slam_v44/pose` 作为 fallback。

### Step3：A* 全局路径规划

导航节点在 `/slam_v44/map` 上做 A* 搜索，从机器人当前位置规划到目标点。

规划时还会对障碍物做膨胀，避免机器狗贴着墙或障碍物走。

### Step4：本地控制

规划出路径后，控制器会在路径上选择一个前视点，计算当前朝向和目标方向的角度误差，然后输出：

```text
线速度 + 角速度
```

同时导航节点会订阅雷达，只看正前方一定角度范围内有没有障碍物。如果前方障碍物太近，就停止；如果只是比较近，就减速。

## 六、导航问题的修正

原版导航经常失败，是因为机器人全局位姿和地图可能不稳定，导致 A* 看到的地图不是清晰的教室结构，而是一堆错误叠加的障碍物。这样目标点容易被判断为不可到达，或者系统误以为机器人已经在目标附近。

v44 加入 odom 后，机器人位姿会随着真实运动连续更新，地图也更容易稳定在同一个 `map` 坐标系里。因此导航的起点、目标点、障碍物地图三者更容易处在同一个正确坐标系下，A* 规划成功率应该比原版高。

但导航仍然依赖几个条件：

```text
/slam_v44/map 地图要稳定
/slam_v44/pose 位姿要连续变化
TF 要正常
目标点要在地图范围内
目标点不能落在障碍物或未知区域
twist_sub 要先启动
真正运动时 cmd_vel_topic 要切到 /cmd_vel
```

目前红点比较杂乱，更多是持久障碍物显示层太敏感，不一定代表地图本体完全错误。判断导航能不能用，应该优先看 `/slam_v44/map`、`/slam_v44/pose` 和 TF 是否稳定。
